# uNews 0.3.5

Текущая версия: `0.3.5` — восстановление реальной очереди и гарантированная фиксация состояния при сбоях.

## Что изменено

- `publish:all` запускается через защитный runner `scripts/run-publish-all-news.js`.
- При фатальном сбое реального запуска безопасная причина записывается в `data/health.json` и `data/errors.json`.
- Токены и token-like строки удаляются из сохраняемого текста ошибки.
- Предыдущий `last_successful_check_at` не подменяется временем неуспешной попытки.
- Успешный запуск записывает `last_attempt_status: success` и очищает `last_error`.
- Telegram preflight больше не обрывает workflow до сохранения состояния.
- `data/` коммитится до того, как workflow окончательно помечается неуспешным.
- После успешного запуска служебный Issue `uNews queue requires attention` закрывается автоматически.
- Тесты очереди и исходников выполняются при каждом реальном запуске, включая расписание.
- FIFO, лимит 20 постов, пауза 61 секунда и немедленный checkpoint после каждого Telegram-поста не изменены.

## Проверенный исходный инцидент

Перед патчем:

- `npm test` прошёл;
- dry-run обнаружил 8 готовых uDream-патчноутов без ошибок;
- Telegram `getMe` и `getChat` прошли для `@uNewsDev_bot` и `@uNewsLog`;
- trigger-коммит не дал новых checkpoint-коммитов;
- `data/health.json` и `data/errors.json` остались устаревшими, потому что workflow завершался до шага commit state.

Версия `0.3.5` исправляет именно потерю диагностического состояния и порядок завершения workflow.

## Точки восстановления

Предыдущее стабильное состояние до автоматизации сохранено в ветке:

```text
stable/manual-publishing-v0.1.0
release/v0.1.0-stable-manual
```

Эти ветки не участвуют в автоматической публикации.

Версии `0.2.1`, `0.2.2`, `0.2.3`, `0.3.0`, `0.3.1`, `0.3.2`, `0.3.3`, `0.3.4` и `0.3.5` фиксируются отдельными release-ветками как стабильные точки восстановления. Автоматический workflow работает только из `main`.
