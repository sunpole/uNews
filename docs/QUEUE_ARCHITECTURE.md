# Очередь uNews

## Назначение

uNews автоматически просматривает все публичные репозитории `sunpole`. Приватные репозитории не запрашиваются и не публикуются.

## Порядок обработки

1. В проекте появляется `news/*.md` с изображением.
2. Патчноут получает обязательные `version` и `queued_at`.
3. uNews проверяет текст, ссылку, картинку и безопасность.
4. Для каждого проекта выбирается самая ранняя версия.
5. Среди готовых проектов выбирается запись с самым ранним `queued_at`.
6. За запуск публикуется ровно одна новость.
7. Сразу после ответа Telegram результат атомарно записывается в `data/published.json`.
8. Workflow коммитит состояние до завершения запуска.

Между двумя успешными публикациями действует глобальная пауза не меньше девяти минут, даже если ручной и scheduled-запуски оказались рядом.

Если ранняя новость проекта ошибочна, следующие версии этого проекта блокируются. Другие проекты продолжают обслуживаться.

## Расписание

Расписание адаптируется к времени с момента последнего нового или изменённого патчноута:

| Простой | Режим | Cron |
|---|---|---|
| меньше 1 часа | каждые 15 минут | `7,22,37,52 * * * *` |
| 1–4 часа | каждые 2 часа | `7 */2 * * *` |
| 4–12 часов | каждые 12 часов | `7 */12 * * *` |
| 12–24 часа | раз в сутки | `7 7 * * *` |
| 24–72 часа | примерно раз в 3 дня | `7 7 */3 * *` |
| от 72 часов | раз в неделю | `7 7 * * 1` |

Готовая очередь всегда удерживает быстрый режим, пока не будет обработана. Одинаковый ошибочный патчноут не считается новой активностью на каждом проходе; исправление файла меняет Git SHA и снова пробуждает очередь.

Текущее состояние хранится в `data/scheduler.json`. uNews изменяет только помеченную cron-строку своего workflow и фиксирует изменение вместе с состоянием. Пустой проход не создаёт коммит; недельный heartbeat сохраняет активность публичного scheduled workflow.

Цена экономии — задержка обнаружения после длительного простоя может достигать недели. Для срочной новости используется ручной реальный запуск `workflow_dispatch`: новая ревизия патчноута немедленно переводит систему в быстрый режим.

## Контроль

- последний запуск берётся из GitHub Actions;
- состояние очереди хранится в `data/health.json`;
- ошибки хранятся в `data/errors.json`;
- при сбое или заблокированном проекте создаётся один GitHub Issue;
- после восстановления служебный Issue закрывается автоматически.

## Секреты

Перед каждым реальным запуском выполняются безопасные `getMe` и `getChat`. Значения токена и идентификатора канала не печатаются. Telegram bot token не имеет заданного календарного срока: действительность проверяется ответом API.
