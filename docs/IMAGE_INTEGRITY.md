# Проверка изображений uNews

## Назначение

uNews обязан обнаружить повреждённое или подменённое изображение **до** обращения к Telegram. Наличие файла по URL и правильное расширение больше не считаются достаточным доказательством.

Версия `0.3.7` вводит единый контракт image integrity для:

- полного сканирования всех ожидающих патчноутов;
- повторной проверки непосредственно перед публикацией;
- локальной проверки одного патчноута;
- одиночных изображений и Telegram-альбомов.

## Причина появления проверки

В очереди uDream находился файл с расширением `.png`, доступный по правильному URL. Простая проверка `HEAD` проходила, но Telegram возвращал `IMAGE_PROCESS_FAILED`.

Глубокая диагностика показала:

- PNG-сигнатура была правильной;
- размер изображения читался как `600×315`;
- chunk `PLTE` имел неверный CRC;
- `pngcheck`, ImageMagick и Telegram не могли корректно декодировать файл.

Позже Telegram принял исправленный пост `uDream 23.8.0` как message `54`; запись сохранена в `data/published.json`, поэтому повторная публикация исключена.

## Канонический путь

```text
news/*.md
    ↓
policy текста и YAML
    ↓
GET каждого изображения
    ↓
проверка bytes, формата и структуры
    ↓
FIFO selection
    ↓
повторный GET непосредственно перед отправкой
    ↓
проверенный Buffer → Blob
    ↓
Telegram sendPhoto / sendMediaGroup
    ↓
message_id → atomic state → Git checkpoint
```

Telegram больше не получает сырой URL изображения. uNews скачивает файл сам, проверяет его и передаёт Telegram уже проверенные bytes как multipart Blob.

## Два обязательных уровня

### 1. Полный аудит очереди

`scripts/publish-all-news.js` проверяет каждое изображение каждого неопубликованного кандидата.

Результат:

- все повреждённые pending-записи видны за один dry-run;
- ошибка ранней версии блокирует только следующие версии того же проекта;
- другие проекты продолжают участвовать в очереди;
- в Telegram ничего не отправляется, пока запись не прошла проверку.

### 2. Повторная проверка перед Telegram

Выбранный патчноут скачивается повторно непосредственно перед отправкой.

Это защищает от ситуации, когда файл изменился между сканированием и публикацией. В Telegram передаются bytes именно второй успешной проверки.

## Поддерживаемые форматы

Разрешены:

- `.png` → PNG;
- `.jpg`, `.jpeg` → JPEG;
- `.gif` → GIF;
- `.webp` → WebP.

Расширение обязано совпадать с фактическими bytes. Например, PNG под именем `image.jpg` блокируется.

## Общие ограничения

По умолчанию:

- файл не может быть пустым;
- файл не может превышать 20 MiB;
- ширина и высота должны быть положительными;
- отдельная сторона не может превышать 20 000 px;
- соотношение сторон не может превышать 20:1;
- неизвестный формат блокируется.

## Глубокая проверка PNG

Для PNG проверяются:

- восьмибайтовая PNG-сигнатура;
- целостность заголовка каждого chunk;
- допустимое ASCII-имя chunk;
- CRC каждого chunk;
- первый chunk строго `IHDR`;
- единственный корректный `IHDR` длиной 13 байт;
- допустимая комбинация `bit depth` и `color type`;
- стандартные compression/filter/interlace methods;
- корректный `PLTE`, когда он нужен;
- наличие непустого `IDAT`;
- успешное zlib-декодирование объединённых `IDAT` bytes;
- обязательный пустой `IEND`;
- отсутствие любых trailing bytes после `IEND`.

Indexed PNG (`color type 3`) без `PLTE` блокируется.

## Проверка JPEG, GIF и WebP

JPEG:

- `SOI` и `EOI` markers;
- корректные длины сегментов;
- наличие SOF-сегмента и размеров.

GIF:

- сигнатура `GIF87a` или `GIF89a`;
- корректные размеры logical screen;
- trailer `0x3B`.

WebP:

- контейнер `RIFF`;
- сигнатура `WEBP`;
- точное совпадение RIFF declared size с размером файла.

## Локальная проверка

Проверить один патчноут без Telegram:

```bash
npm run publish:projects:check -- "../project/news/example.md"
```

Проверить всю очередь:

```bash
npm run publish:all:check
```

Запустить все регрессионные проверки:

```bash
npm test
```

`npm test` включает `scripts/check-image-integrity.js`, который проверяет:

- корректный indexed PNG;
- реальный сценарий неверного CRC `PLTE`;
- корректный CRC при повреждённом zlib-stream;
- несовпадение расширения и bytes;
- trailing bytes после `IEND`;
- обязательный remote GET;
- HTTP failure;
- лимит `Content-Length`.

## Формат ошибки

Ошибка содержит label патчноута и имя изображения, например:

```text
sunpole/udream|main|news/example.md image example.png: PNG CRC mismatch in PLTE chunk
```

Токены и secrets в сообщение не включаются.

## Поведение очереди при ошибке

- повреждённая запись не публикуется;
- более новые версии того же проекта блокируются FIFO-правилом;
- другие проекты могут продолжить работу;
- ошибка попадает в структурированный отчёт запуска;
- при фатальном сбое recovery-runner сохраняет безопасную причину в `data/health.json` и `data/errors.json`;
- workflow сначала фиксирует state, затем возвращает неуспешный статус.

После исправления источник должен пройти новый полный dry-run. Только затем разрешается реальный запуск.

## Ремонт изображений

Ещё не опубликованный патчноут ремонтируется в проекте-источнике с сохранением:

- `project`;
- `series`;
- `version`;
- `queued_at`.

Уже опубликованный Telegram-пост нельзя чинить повторной публикацией. Используются maintenance-команды по сохранённому `message_id`:

```bash
npm run edit:media -- -- --message-id 54 --patchnote "../project/news/example.md" --key "published-key" --record-state
npm run edit:caption -- -- --message-id 54 --patchnote "../project/news/example.md" --key "published-key" --record-state
```

## Постоянные файлы реализации

- `scripts/lib/image-integrity.js` — byte-level валидация и remote GET;
- `scripts/check-image-integrity.js` — регрессионные fixtures;
- `scripts/publish-all-news.js` — аудит очереди и повторная проверка перед Telegram;
- `scripts/publish-from-projects.js` — локальная проверка одного патчноута;
- `scripts/lib/telegram-client.js` — multipart-отправка проверенного Blob;
- `docs/QUEUE_ARCHITECTURE.md` — FIFO и checkpoint lifecycle.

## Запрещено

- возвращать проверку только через `HEAD`;
- доверять одному расширению файла;
- передавать Telegram непроверенный raw URL;
- игнорировать CRC или zlib error PNG;
- публиковать более новую версию поверх заблокированной ранней версии проекта;
- хранить временные diagnostic workflow и отчёты в стабильном `main`;
- выводить Telegram token, GitHub token или значения secrets в лог.
